# Card Naming and IP Guidelines

Cyber Realm card names, rules text, flavor text, and public card files should avoid unnecessary use of proprietary, trademarked, or brand-specific names.

This is a creative and production guideline, not legal advice.

## Core rule

Use generic, descriptive, fictional, or Cyber Realm-original names unless a real name is required for educational context and cleared for use.

## Avoid using brand names as card names

Do not create cards named directly after commercial products, vendors, platforms, or tools unless the use has been reviewed and approved.

Prefer generic training names.

Examples:

| Avoid | Prefer |
|---|---|
| Shodan Search | Internet Exposure Search |
| Splunk Dashboard | SIEM Dashboard |
| Palo Alto Firewall | Next-Gen Firewall |
| Fortinet VPN | Vendor VPN Gateway |
| TeamViewer Access | Remote Support Session |
| SolarWinds Compromise | Trusted Update Compromise |

## Safer naming patterns

Use names based on:

- attacker behavior
- defender action
- asset function
- operational consequence
- training lesson
- fictional in-world Cyber Realm terminology

Examples:

- Default Login Attempt
- Misconfigured Remote Access
- Unsupported HMI
- Legacy Engineering Workstation
- Vendor VPN Gateway
- Internet Exposure Search
- Credential Reuse
- Noisy Malware Drop
- Trusted Update Compromise
- Manual Override
- Network Segmentation

## Review checklist

Before a card moves from Draft to Final, check:

- Does the card name include a company, product, malware, or vendor name?
- Does the flavor text mention a proprietary name?
- Could the card be understood using a generic name instead?
- Is the name original enough to support Cyber Realm branding?

## Preferred default

When in doubt, rename the card to a generic Cyber Realm-style term and keep real-world references in instructor notes or private production material.
